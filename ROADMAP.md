## 1. Аудит текущего состояния (ИСХОДНЫЙ — большинство гэпов закрыто, см. статус выше)

Построено и стыковано (15 Dart-файлов + 4 нативных + setup-гайд):

| Слой | Файлы | Состояние |
|---|---|---|

| Синхронизация | `mesh_sync_engine.dart`, `crdt_materializer.dart` | Готов |

| UI | `app_theme.dart`, `dashboard_view.dart`, `mesh_ui_adapter.dart` | Готов |
| Composition root | `main.dart` (+ композер, fallback-эмбеддинги) | Готов |

Честный гэп-анализ — чего НЕТ:

1. **Ни одна строка не прошла через компилятор.** Код написан compile-safe по памяти API; первый `flutter analyze` + сборка нативных таргетов неизбежно даст пачку мелких правок (генерируемые имена Isar, сигнатуры Nearby/Multipeer).
2. **Онбординг разрешений** — location/Bluetooth/Nearby permissions запрашивать некому; без них транспорт и телеметрия молча fail-closed.
3. **QR-пейринг** Light Client ↔ Core Node (сервер готов, UI генерации/сканирования QR — нет).
4. **Протокол подтверждения кольца не замкнут**: lock-опы гоняются, но нет UX «все N сторон залочили → кольцо подтверждено → satisfy_intent» и уведомлений.
5. **Реальные эмбеддинги**: FNV-1a — тестовый детерминированный суррогат; продакшен-качество мэтчинга требует ONNX MiniLM (int8, ~23 МБ).
6. **Module B — только гейт**, нет воркера (дистрибуция чанков, ONNX runtime, proof-of-computation).
7. **Module C — только UI-плитки**, нет Modbus TCP/MQTT-шлюза.
8. **Тестов ноль.** Для CRDT-системы это недопустимо дольше одной недели.
9. **Стор-комплаенс**: PrivacyInfo.xcprivacy, Play Data Safety, тексты для App Review (Multipeer/Local Network — частый триггер вопросов ревью).
10. **Репутация** (`reliabilityScore`) нигде не вычисляется.

## 2. Фазированный план

Оценки — для одного разработчика полной занятости. Правило приоритета: **FluidMesh (Module A) — единственный модуль в v1.0**; B и C идут пост-релизом. Один модуль, доведённый до конца, побеждает три полуживых.

### Фаза 0 — «Компилируется и запускается» (1 неделя)
- `flutter create` каркас, перенос файлов, pubspec (isar, cryptography, web_socket_channel, flutter_secure_storage, path_provider), `build_runner`.
- Чинить analyzer/компиляцию Dart + Kotlin + Swift до нуля ошибок и warning'ов.
- Экран онбординга: запрос разрешений (permission_handler), выбор алиаса, показ собственного публичного ключа.
- **Выход:** приложение стартует на физическом iPhone и Android, статус-строка живёт, телеметрия читается.

### Фаза 1 — E2E FluidMesh (2–3 недели)
- Двухдевайсный тест: intent → gossip → ring → lock на паре Android↔Android и iOS↔iOS.
- QR-пейринг: Core Node рендерит `ws://<ip>:7411` в QR; web-клиент открывается с `?bridge=`.
- Замыкание протокола: экран «кольцо подтверждено» (все hop'ы залочены по CRDT-логу), действие «выполнено» → `satisfy_intent`, локальные уведомления.
- Вычисление `reliabilityScore` из подписанной истории satisfied-колец.
- Тесты: unit (детерминизм матчера, фолд материализатора, преймидж/подписи, causalCompare), golden — UI.
- **Выход:** три телефона в одной комнате закрывают 3-стороннее кольцо без интернета.

### Фаза 2 — Реальный интеллект (2 недели)
- ONNX Runtime (пакет `onnxruntime` или FFI): all-MiniLM-L6-v2 int8 за интерфейсом `EdgeInferenceService`. Ничего выше слоя не трогается — интерфейс проектировался под эту замену.
- Токенизатор (WordPiece) на Dart, модель в ассетах, warm-up на сплэше.
- Порог `similarityThreshold` перекалибровать на реальных векторах (0.45 для MiniLM — стартовая точка, валидировать на корпусе из ~50 реальных объявлений).
- **Выход:** «offer: репетитор Dart» ↔ «need: помочь с флаттером» мэтчится, мусор — нет.

### Фаза 3 — Релизная инженерия (2 недели)
- CI: GitHub Actions (analyze, test, сборки), Fastlane для подписи и выгрузки.
- iOS: PrivacyInfo.xcprivacy, review notes про Multipeer/Local Network («приложение работает офлайн, серверов нет» — писать явно, иначе ревью задаёт вопросы), TestFlight.
- Android: Play Data Safety («данные не покидают устройство»), closed testing track, плюс подписанный APK для прямой раздачи с сайта.
- Диагностика без серверов: локальный debug-экран (лог материализатора, счётчики rejected-подписей), опционально Sentry free tier с opt-in.
- Юридика: Privacy Policy и ToS (обязательны для обоих сторов; для local-first продукта они короткие и честные — данные не собираются).
- **Выход:** билд в TestFlight и в closed track, 10–20 внешних тестеров.

### Фаза 4 — Запуск (1 неделя)
- Сайт (см. §4) на Cloudflare Pages/Vercel, PWA как отдельный деплой того же web-билда.
- Полевой тест в целевой локации (коворкинг): 10+ устройств, замер времени закрытия первого кольца.
- **Выход:** публичный релиз v1.0 = FluidMesh.

### Фаза 5+ — Пост-релиз (по 3–4 недели на модуль)
- **Module B:** воркер (ONNX-инференс чанков, очередь задач через те же CRDT-опы, proof-of-computation = подпись хэша выхода), UI очереди в COMPUTE-вкладке.
- **Module C:** Modbus TCP клиент (пакет `modbus_client`) + MQTT (`mqtt_client`), маппинг регистров Deye/Growatt, реле Shelly через локальный HTTP API.
- **Кросс-OS радио:** кастомный BLE GATT транспорт за `LocalMeshTransportService` (снимает ограничение Multipeer↔Nearby).
- Web-персистентность: Drift-wasm бэкенд за `MeshRepository`.

## 3. Дистрибуция — жёсткие факты

- **iOS: прямой скачки с сайта не существует.** Кнопка на сайте ведёт только в App Store (или TestFlight на бете). В ЕС есть alt-store'ы по DMA, но для соло-разработчика это не стартовый путь.
- **Android:** три канала — Google Play, **прямой APK с сайта** (подпись release-ключом, инструкция про «неизвестные источники»), позже F-Droid (идеален для open-source local-first позиционирования, но требует reproducible builds).
- **Web PWA:** кнопка «Launch» → задеплоенный web-билд; работает как Light Client при наличии Core Node в сети.

## 4. Сайт (`index.html` — готов, лежит рядом)

- **Стек:** один самодостаточный статический HTML, ноль внешних запросов (system-шрифты, inline CSS/SVG) → LCP < 1 с, CLS 0, идеальные Core Web Vitals из коробки. Хостинг: Cloudflare Pages или Vercel, $0.
- **Дизайн:** тот же язык, что у приложения — obsidian, 1px slate, zero-radius, цвет только как 2px информационные штрихи. Сигнатурный элемент — SVG-диаграмма замкнутого бартерного кольца (ядро продукта, не декор).
- **GEO/SEO:** JSON-LD `SoftwareApplication` + `FAQPage`, семантическая иерархия h1→h3, FAQ в `<details>` (парсится и людьми, и LLM-краулерами), canonical/OG/Twitter meta. Рекомендую добавить `/llms.txt` с конспектом продукта при деплое.
- **Перед публикацией заменить:** ссылки сторов (плейсхолдеры `#`), домен в canonical/OG, e-mail в футере.
